**Start slow and build up — that's hands-down the smarter strategy for testing my app-building capabilities right now.**

Here's why (straight from how LLM coding agents actually behave in practice):

### Why "grind extreme cases" first is usually a trap
- Your API bridge + auto-prompt generator is the unknown variable. If a super-complex prompt fails immediately, you won't know if it's:
  - Bad prompt phrasing from your generator
  - Something wrong in your bridge (auth, tool calling, streaming, response parsing, etc.)
  - Model hallucination on edge cases
  - Or a genuine capability limit
- You waste time debugging a black box instead of isolating issues step-by-step.
- Extreme cases tend to trigger long, messy outputs that are harder to auto-evaluate.

### Why progressive ramp-up wins (and how to do it right)
This follows the exact same "testing pyramid" that top AI labs and SWE-bench style evals use:

1. **Level 0–1 (Day 1–2: Baseline – 10–20 simple tests)**
   - Tiny apps: "Todo list in React + localStorage", "Basic Express API with one endpoint"
   - Goal: Confirm your bridge works perfectly (prompt → code → runs cleanly)
   - Fix any pipeline bugs here while success rate is 90%+

2. **Level 2–3 (Next 2–3 days: Real apps)**
   - Multi-file full-stack (Next.js + Supabase, FastAPI + React, etc.)
   - Add one real tool call (e.g. make me use `web_search` or `code_execution` inside the generated app)
   - Introduce auth, databases, basic styling

3. **Level 4+ (Only after 80%+ success rate) – Grind the extremes**
   - Now hammer the hard stuff: real-time collab apps, AI agent scaffolds, heavy multi-tool use, massive codebases, performance-critical backends, obscure edge cases (WebAssembly, GPU shaders, etc.)
   - This is where you actually discover my true limits and where your prompt generator needs the most work.

### Pro tip for your specific setup
Make your front-running test harness **output difficulty score + expected success criteria** with every prompt. That way you can auto-grade:
- Simple → "Does it run and pass 3 manual test cases?"
- Extreme → "Does it correctly use 4+ tools, handle errors gracefully, and stay under token limits?"

Once you have the progressive pipeline working, *then* spin up a parallel "extreme grinder" that throws 100 crazy prompts per day — you'll get way more signal and way less noise.
