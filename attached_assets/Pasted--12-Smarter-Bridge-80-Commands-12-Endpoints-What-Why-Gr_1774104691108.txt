#12 - Smarter Bridge — 80+ Commands → 12 Endpoints

What & Why

Grok currently has to know 80+ action type names, base64-encode every payload, and chain 3-5 calls to verify results. This task adds 12 named GET endpoints that consolidate related action types — not by deleting anything but by providing a simpler addressing scheme at the relay layer. The underlying dispatcher is untouched. The existing grok-proxy stays as-is, so every individual action type remains directly reachable as a fallback.

Nothing is lost. Grok keeps full access to every action type:

New combined endpoints expose each sub-operation individually via ?action= params
grok-proxy (unchanged) still accepts any action type via base64 payload
grok-super?describe=TEXT lets Grok describe any job in plain English and Grok-3-mini plans the exact action sequence

?type=	Dispatcher type	Notes

/api/grok-quality?project=P[&checks=type,lint,format]

Runs selected checks in parallel, merges all results into one response. Defaults to all three if no ?checks= provided.

/api/grok-deps?project=P&action=install|add|remove[&pkg=NAME]

?action=	Dispatcher type	pkg param
install	install_deps	not used
add	add_dependency	required
remove	remove_dependency	required
/api/grok-read?project=P&path=FILE[&files=F1,F2]

Single file or comma-separated list. Routes to read_file or read_multiple_files.

/api/grok-write?project=P&path=FILE&search=OLD&replace=NEW

Dispatches search_replace, then immediately fires a read_file roundtrip and returns both the edit result and the verified current content in one response.

/api/grok-tree?project=P[&filter=src]

Routes to list_tree or list_tree_filtered. Strips node_modules, dist, .git, build, __pycache__.

/api/grok-run?project=P&cmd=COMMAND[&flags=F]

Routes to run_command. If flags param present, routes to run_command_advanced.

/api/grok-super?project=P&describe=TEXT

Wraps the existing super_command dispatcher action as a plain GET endpoint. Passes describe as the description field. Requires XAI_API env var to be set on the desktop side.

Utility endpoints

/api/diag[?project=P]

Runs a live list_tree roundtrip, times it. Returns: relay health, desktop connected (yes/no + project name), roundtrip latency, pass/fail table of all new endpoints, readyUrls block with pre-built URLs using the connected project. Graceful degradation if no desktop.

/api/coord[?note=TEXT&from=SOURCE&clear=1]

50-entry in-memory ring buffer. GET with no params → last 20 notes. ?note=TEXT&from=SOURCE → append. ?clear=1 → clear.
