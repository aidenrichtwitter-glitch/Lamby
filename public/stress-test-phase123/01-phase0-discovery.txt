STEP 4: Architecture confirmed
  Request Chain:
    1. Build payload: base64(JSON.stringify({actions:[{type,project,...}]}))
    2. GET /api/grok-proxy?project=P&payload=ENCODED_BASE64
    3. Relay decodes, forwards via WebSocket to desktop connector
    4. Desktop executes (file I/O, git, shell, etc.)
    5. Result returns over WebSocket → relay → HTTP response

  Key Constraints Identified:
    - Grok browse_page = GET only, no POST
    - URL length: relay accepts 11KB+, but browse_page may truncate ~8KB
    - WebSocket timeout: 30 seconds
    - Desktop runs Windows (PowerShell for shell commands)
    - Serial command queue (one connection, no parallelism)

STEP 5: Critical bug discovered
  BUG: "project" field must be inside each action object, NOT just query param
  Without it: files land in wrong directory, unfindable by grok-read
  CORRECT: {"actions":[{"type":"write_file","project":"groks-app","path":"..."}]}
  WRONG:   {"actions":[{"type":"write_file","path":"..."}]} + ?project=groks-app

STEP 6: Encoding requirement discovered
  Base64 output contains + and = characters
  These break URL query params if not encoded
  SOLUTION: Always encodeURIComponent(btoa(json)) before putting in payload= param
  Without this step: silent failures, corrupted payloads
