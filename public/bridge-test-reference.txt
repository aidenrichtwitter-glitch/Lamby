Architecture: Lamby Bridge Relay - zero-dependency WebSocket bridge between Grok (cloud AI) and a local Windows desktop running Node.js.

Request Chain:
  1. Grok builds payload = base64(JSON.stringify({actions:[{type,project,...}]}))
  2. Grok issues HTTP GET /api/grok-proxy?payload=PAYLOAD to this relay (Replit cloud)
  3. Relay decodes payload, forwards actions via WebSocket to the connected desktop connector
  4. Desktop connector (Node.js on Windows) executes: file I/O, shell commands, git, etc.
  5. Connector sends result back over WebSocket
  6. Relay returns JSON to Grok: {success:true, results:[{type:'command_type', data:{...}}]}

Constraints:
  - Grok is GET-only - every request must be a GET URL. POST is not available.
  - URL length limit: Relay accepts URLs up to at least 11KB. However, Grok's browse_page tool may truncate URLs around ~8KB. Safe rule for Grok: keep content under ~5KB per write_file. Use write_file_chunk for larger.
  - All commands are async over WebSocket - relay waits up to 30s for desktop response.
  - Desktop connector runs on Windows - but run_command has a whitelist (see Section 2).
  - One WebSocket connection per desktop instance - commands queue; do not fire many in parallel.

Project Resolution: The connector resolves project names to disk paths. Given project='groks-app', it looks for: PROJECT_DIR/projects/groks-app then PROJECT_DIR/groks-app. File paths in commands are relative to the resolved project root.

Dev Server Detection: The connector dynamically detects the Vite/Next/CRA dev server port by scanning active TCP listeners on ports 3000-9999 using netstat. Result cached for 2 minutes. Use get_preview_url to get the live URL.

HMR: Vite uses Hot Module Replacement - writing a file via write_file or write_file_chunk triggers an instant browser update with no server restart needed, ONLY if you write to the correct file (the one actually imported by index.tsx).

SECTION 2: CRITICAL RULES (MUST FOLLOW) — STRESS-TESTED
=====================================

1. PROJECT IN ACTIONS (BUG FIX): The "project" field MUST be inside each action object in grok-proxy payloads:
   CORRECT: {"actions":[{"type":"write_file","project":"groks-app","path":"...","content":"..."}]}
   WRONG: {"actions":[{"type":"write_file","path":"...","content":"..."}]} (project only in query param)
   Without it, files land in the wrong directory and are unfindable by grok-read.

2. ENCODING (CRITICAL): When building grok-proxy URLs:
   step1: JSON.stringify({actions:[...]})
   step2: btoa(jsonString)
   step3: encodeURIComponent(base64String) ← THIS STEP IS CRITICAL
   step4: PUT in URL: /api/grok-proxy?project=P&payload=ENCODED
   WHY: Base64 contains + and = characters. Without encodeURIComponent, these break the URL and cause silent failures.

3. URL SIZE LIMITS (TESTED):
   The relay itself accepts URLs up to at least 11KB (tested: 10920-char URL worked fine).
   However, Grok's browse_page tool has its own URL length limit (~8KB).
   Safe rule for Grok: keep write_file content under ~5KB. For larger files, use write_file_chunk.
   TESTED RESULTS (relay-side, no Grok browse_page involved):
     4KB content → 5594-char URL → PASS (verified content intact)
     5KB content → 6930-char URL → PASS (verified content intact)
     6KB content → 8250-char URL → PASS (verified content intact)
     8KB content → 10920-char URL → PASS (verified content intact)

4. WRITE_FILE_CHUNK: For large files, split into ~1500-char chunks. Send chunk_index 0..N-1, total_chunks=N. One action per chunk. File assembles and writes only when final chunk arrives. Add 100-150ms delay between chunk requests.
   TESTED: 6021 chars → 5 chunks → all 5 succeeded → verified 6021 chars written intact (1052ms total).

5. ENTRY POINT: Before writing App.tsx (or any root component) ALWAYS read src/index.tsx first. Projects often import from 'components/App' not 'App', so the real file is src/components/App.tsx. Writing to src/App.tsx when nothing imports it has zero visible effect.

6. DELETE SAFETY: ALWAYS check what imports a file before deleting it. A single delete_file that removes a file still imported by App.tsx will crash the ENTIRE application. Read the importing file first, update imports or write a replacement BEFORE deleting.
   TESTED: Deleting Analytics.tsx while App.tsx still imported it caused Vite error overlay crash. Had to restore it.

7. SEARCH_REPLACE ZERO MATCHES: If grok-write returns {replacements:0}, the exact string was NOT found. Response format: {"results":[{"path":"file","replacements":0}]}. Read the file first, copy exact text, retry.
   TESTED: Searched for "NONEXISTENT_STRING_xyz" → got replacements:0.

8. GIT RESET LIMITATION: git reset --hard is sanitized to --soft by the bridge connector. You CANNOT do destructive resets. To truly revert, manually restore files via write_file/delete_file.
   TESTED: Sent reset --hard HEAD~1, got --soft HEAD executed instead.

9. RUN_COMMAND WHITELIST: The run_command endpoint (grok-run) has a command whitelist.
   ALLOWED: npm (TESTED: "npm --version" → "11.7.0"), node (TESTED: node -e "console.log(1+1)" → 2), echo (TESTED: "echo test123" → "test123")
   BLOCKED: dir, cd, type, Get-ChildItem, ls (TESTED: "dir src" → "Command not allowed")
   Use grok-tree for file listing and grok-read for file reading instead.
   NOTE: Param name is "command" (not "cmd").

10. VERIFY AFTER EVERY EDIT: ALWAYS use get_console_errors after edits to check for broken imports/compile errors. TESTED: returns {"errors":[],"totalErrors":0} when healthy.

SECTION 3: HOW TO MAKE REQUESTS (grok-proxy)
=====================================

Endpoint: /api/grok-proxy
Method: GET
Params: project=PROJECT_NAME & payload=base64(JSON.stringify({actions:[{type, project, ...params}]}))

Encoding Steps (VERIFIED WORKING):
  step1: Build actions array: [{type:'write_file', project:'groks-app', path:'src/newfile.tsx', content:'...'}]
  step2: JSON.stringify the wrapper: JSON.stringify({actions:[...]})
  step3: base64 encode: btoa(step2)
  step4: URL-encode the base64: encodeURIComponent(step3)  ← MANDATORY
  step5: GET /api/grok-proxy?project=groks-app&payload=ENCODED_BASE64

Auto Repair: Relay auto-fixes common encoding mistakes: escapes raw control chars, strips trailing commas, accepts 'types'/'commands'/'data'/'payload' as aliases for 'actions'.
Batch Actions: You can chain up to 10 actions in one request. All execute sequentially on the desktop.
